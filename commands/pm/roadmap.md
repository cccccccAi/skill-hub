# 路线图管理框架

## 路线图格式

### Now / Next / Later（推荐）

| 时间段    | 含义          | 内容                 |
| --------- | ------------- | -------------------- |
| **Now**   | 当前迭代      | 已确认需求，正在开发 |
| **Next**  | 下 1-2 个迭代 | 已评审，准备开发     |
| **Later** | 未来 1-3 个月 | 方向确定，细节待定   |

**优点：** 不绑定具体日期，避免被日期追着跑
**适合：** 敏捷团队、需求变化快的产品

### 季度路线图

按 Q1/Q2/Q3/Q4 划分，每个季度 2-3 个主题：

```
Q1：用户增长
  - [功能 A] — Must
  - [功能 B] — Should

Q2：商业化
  - [功能 C] — Must
  - [功能 D] — Could
```

**适合：** 向管理层/投资人汇报

### OKR 对齐路线图

```
O: [目标]
  KR1: [关键结果] → 支撑功能：[功能列表]
  KR2: [关键结果] → 支撑功能：[功能列表]
```

**适合：** OKR 驱动的组织

## 优先级排序框架

### RICE 评分

| 维度           | 含义             | 评分方式                        |
| -------------- | ---------------- | ------------------------------- |
| **R**each      | 影响多少用户     | 预估用户数/季度                 |
| **I**mpact     | 对单个用户的影响 | 3=大 2=高 1=中 0.5=低 0.25=极低 |
| **C**onfidence | 估算的置信度     | 100%=高 80%=中 50%=低           |
| **E**ffort     | 工作量           | 人月数                          |

**公式：** RICE = (Reach × Impact × Confidence) / Effort

**技巧：**

- Confidence 低于 50% 的需求，先做用户验证再排序
- 不要对所有需求都打高 Impact——强制区分

### MoSCoW 分类

| 级别   | 占比建议 | 判断问题                 |
| ------ | -------- | ------------------------ |
| Must   | ≤ 60%    | 没有它能上线吗？         |
| Should | 20%      | 对体验有明显影响吗？     |
| Could  | 15%      | 有时间就做，没时间不影响 |
| Won't  | 5%       | 明确不做，写下原因       |

### ICE 评分

| 维度           | 含义     | 评分              |
| -------------- | -------- | ----------------- |
| **I**mpact     | 影响程度 | 1-10              |
| **C**onfidence | 置信度   | 1-10              |
| **E**ase       | 实施难度 | 1-10（10=最容易） |

**公式：** ICE = Impact × Confidence × Ease

**适合：** 快速粗筛，不适合精细排序

### 价值-工作量矩阵

```
        高价值
          │
  Quick   │   Big
  Wins    │   Bets
──────────┼──────────
  Fill    │   Money
  Ins     │   Pit
          │
        低价值
  低工作量      高工作量
```

| 象限       | 策略                 |
| ---------- | -------------------- |
| Quick Wins | 优先做，低成本高回报 |
| Big Bets   | 仔细评估，值得投入   |
| Fill Ins   | 有空再做             |
| Money Pit  | 避免，高成本低回报   |

## 依赖管理

**识别依赖类型：**

- 技术依赖：A 功能需要 B 接口先完成
- 数据依赖：需要某个数据源就位
- 团队依赖：需要另一个团队配合
- 外部依赖：等待第三方/审批/政策

**可视化依赖：**

```
[功能 A] ──依赖──→ [功能 B]
    │                  │
    └──依赖──→ [接口 C]──┘
```

## 容量规划（70/20/10）

| 分配 | 用途        | 说明              |
| ---- | ----------- | ----------------- |
| 70%  | 计划内需求  | 路线图上的功能    |
| 20%  | 技术债/优化 | 不做会越欠越多    |
| 10%  | 意外/探索   | 紧急需求 + 小实验 |

**常见错误：**

- 100% 排满计划内需求（没有缓冲）
- 技术债永远排不上（直到系统崩溃）
- 容量估算过于乐观（乘以 1.3-1.5 更接近现实）
