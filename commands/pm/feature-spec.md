# PRD 写作框架

## PRD 结构模板

### 1. 问题陈述

- **背景**：为什么现在要做这件事？
- **痛点**：用户/业务遇到了什么问题？（用数据支撑）
- **机会**：解决后能带来什么价值？

### 2. 目标与非目标

**目标**（3-5 个，具体可衡量）：

- 每个目标用"通过 [方法] 实现 [结果]"的句式
- 目标之间不应冲突

**非目标**（明确不做什么）：

- 主动划定边界，比"目标"更重要
- "如果所有事都是 P0，那就没有 P0"

### 3. 用户故事

**INVEST 原则：**
| 原则 | 含义 | 检查方法 |
|------|------|---------|
| Independent | 独立 | 能否单独交付？ |
| Negotiable | 可协商 | 是否过于具体？ |
| Valuable | 有价值 | 用户/业务能感知到吗？ |
| Estimable | 可估算 | 开发能给出工时吗？ |
| Small | 小粒度 | 一个迭代能完成吗？ |
| Testable | 可测试 | 怎么验证做对了？ |

**格式：** 作为 [角色]，我想要 [功能]，以便 [价值]

**常见错误：**

- 写成技术任务而非用户需求
- 太大无法在一个迭代完成
- 缺少价值说明（"以便"部分）
- 把实现方案写进了故事里

### 4. 需求分级（MoSCoW）

| 级别       | 含义     | 判断标准                   |
| ---------- | -------- | -------------------------- |
| **Must**   | 必须有   | 没有就不能上线             |
| **Should** | 应该有   | 体验显著提升，但缺了也能用 |
| **Could**  | 可以有   | 锦上添花                   |
| **Won't**  | 本次不做 | 明确排除，避免范围蔓延     |

**技巧：** Must 不应超过总需求的 60%。如果超过，说明没有真正做取舍。

### 5. 验收标准

**格式 A：Given/When/Then（推荐）**

```
Given: 用户已登录且有未读消息
When: 用户点击消息图标
Then: 显示消息列表，未读消息标记为蓝色圆点，按时间倒序排列
```

**格式 B：检查清单**

- [ ] 消息列表支持分页加载
- [ ] 单条消息可标记已读
- [ ] 空状态显示"暂无消息"

### 6. 成功指标

**先行指标**（预测未来趋势）：

- 功能激活率、首次使用完成率、日活/周活变化
- 用户反馈评分、NPS 变化

**滞后指标**（验证最终效果）：

- 留存率变化、转化率变化、营收影响
- 客服工单减少量

**设定目标：**

- 成功线：达到即算成功
- 挑战线：超出预期
- 每个指标注明：测量方式 + 评估周期 + 数据来源

### 7. 范围管理

**识别范围蔓延的 5 个信号：**

1. "顺便加一个小功能"
2. "用户可能还需要…"
3. 需求讨论会越开越长
4. 上线时间反复推迟
5. 开发说"需求又变了"

**防止范围蔓延的策略：**

1. 每个需求对照 MoSCoW 分级
2. 新增需求必须替换掉一个同级需求
3. "Won't" 列表和 "Must" 列表同样重要
4. 定期回顾：这个需求是否还服务于最初的目标？
