# 汇报沟通框架

## 核心原则

- **结论先行**：最重要的信息放第一句
- **状态灯不骗人**：黄灯不是失败，是好的风险沟通
- **请求要具体**："我们需要帮助"不是请求，"需要在周五前确定 X 方案"才是

## 按受众模板

### 管理层汇报

**字数限制：300 字以内**

```
## [项目/产品名] [周报/月报] — [日期]

**状态：** 🟢 正常 / 🟡 有风险 / 🔴 需关注

**一句话总结：** [最重要的一件事]

**关键进展：**
- [成果 1]（量化影响）
- [成果 2]

**风险与阻塞：**
- [风险]：[影响] → [应对方案] → [需要的决策/资源]

**下周重点：**
- [计划 1]
- [计划 2]

**需要的支持：** [具体请求 + 截止时间]
```

### 技术团队汇报

```
## [项目] 技术周报 — [日期]

**本周完成：**
- [功能/任务]：[技术方案简述]，影响 [模块]
- PR/MR 链接（如有）

**进行中：**
- [任务]：预计 [完成时间]，[当前进度]%

**技术债/待解决：**
- [问题]：影响范围 [描述]，建议 [方案]

**架构变更：**
- [如有重大变更，说明原因和影响]
```

### 跨部门汇报

```
## [项目] 跨部门同步 — [日期]

**项目目标提醒：** [一句话回顾为什么做这件事]

**各方进展：**
| 团队 | 本周进展 | 下周计划 | 需要协作 |
|------|---------|---------|---------|

**关键里程碑：**
- [里程碑]：原计划 [日期]，当前状态 [正常/延迟]

**协作需求：**
- [A 团队] 需要 [B 团队] 在 [日期] 前提供 [内容]
```

### 客户汇报

```
## [产品/项目] 进展通报

**本期亮点：**
- [用客户能理解的语言描述价值]

**即将推出：**
- [功能]：预计 [时间]，将帮助您 [价值]

**如需了解更多：** [联系方式/文档链接]
```

## G/Y/R 状态灯标准

| 状态 | 含义 | 使用场景 |
|------|------|---------|
| 🟢 绿色 | 按计划推进 | 进度正常，无已知风险 |
| 🟡 黄色 | 有风险但可控 | 有问题但有应对方案，需要关注 |
| 🔴 红色 | 需要帮助 | 已偏离计划，需要决策或资源支持 |

**常见错误：**
- 所有项目永远是绿色（粉饰太平）
- 不敢给黄色（怕被追责）
- 红色出现太晚（问题已经爆发）

## ROAM 风险管理

| 类型 | 含义 | 动作 |
|------|------|------|
| **R**esolved | 已解决 | 记录解决方案，关闭 |
| **O**wned | 有人负责 | 指定责任人 + 截止日期 |
| **A**ccepted | 接受风险 | 记录决策原因，监控 |
| **M**itigated | 已缓解 | 记录缓解措施，降级 |

## 会议纪要模板

```
## [会议名称] — [日期]

**参会人：** [名单]
**时长：** [X 分钟]

**关键决策：**
1. [决策内容]（决策人：[谁]）

**行动项：**
| 事项 | 责任人 | 截止日期 |
|------|--------|---------|

**下次会议：** [时间] / [主题]
```
