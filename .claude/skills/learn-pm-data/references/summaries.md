# 通关总结内容

> 以下验证题和原理图使用教育科技行业示例作为默认场景。核心方法论适用于所有行业。
> 使用真实数据路线的用户，AI 应基于用户产品数据调整验证题场景。

## 第1关：指标体系整理

### 验证题

**问题**：某个 L1 指标（如核心功能使用率）从 65% 降到 58%，作为产品负责人你应该？

- A. 立刻重新设计该功能界面
- B. 先看 L2 诊断指标定位原因，再决定动作（✅ 正确）
- C. 这个下降幅度不大，暂时不管
- D. 增加用户培训频次

**解析**：指标变动时不要急于行动。先用 L2 诊断指标（按用户群/地区/版本拆分）定位问题根源。可能是某个细分群的问题，也可能是某次改版导致。没搞清原因就改界面，很可能改错方向。

### 学习要点

- ✅ Skill 是 AI 的知识库，Command 是你触发的工作流（3 分钟热身）
- ✅ North Star 是核心价值的度量，不是"收入"或"用户数"
- ✅ L1 告诉你健康状况，L2 告诉你"为什么"
- ✅ 指标复盘的核心是"so what"——推动行动，不只是看数字

  ┌──────────┴──────────────────┐

## 第2关：用研综合实战

### 验证题

**问题**：一个发现仅来自 2 份访谈，没有其他数据源佐证。它的性质是？

- A. 确定性结论，可以直接指导产品决策
- B. 假设，需要更多数据验证（✅ 正确）
- C. 无效数据，应该忽略
- D. 定性发现，不需要定量验证

**解析**：2 份访谈提供的是"假设"而非"结论"。要成为可靠发现，需要三角验证——用不同方法（访谈 + 问卷 + 行为数据）、不同来源交叉确认。这不意味着要忽略它，而是要标明"置信度：低"，并安排后续验证。

### 学习要点

- ✅ 三角验证（Triangulation）：多源交叉确认才是结论
- ✅ 区分用户"说的"和"做的"——行为数据比口头偏好更可靠
- ✅ 机会评估 = 影响面 × 频率 × 严重度

### ASCII 原理图

```
  定性数据                    定量数据
  ┌──────────┐              ┌──────────┐
  │ 用户访谈 │              │ 行为数据 │
  │ 深度访谈 │              │ 全量用户 │
  └────┬─────┘              └────┬─────┘
       │                         │
       │    ┌──────────┐         │
       └───►│ 三角验证 │◄────────┘
            │Triangulate│
       ┌───►│          │◄────────┐
       │    └────┬─────┘         │
       │         │               │
  ┌────┴─────┐   │         ┌────┴─────┐
  │ 管理者   │   ▼         │ 用户端   │
  │ 问卷调查 │ 发现+置信度 │ 留存队列 │
  └──────────┘             └──────────┘
```

## 第3关：竞品分析

### 验证题

**问题**：竞品分析中最有行动价值（actionable）的部分是？

- A. 功能对比矩阵
- B. 竞品公司介绍
- C. 战略建议（Strategic Implications）（✅ 正确）
- D. 市场份额数据

**解析**：功能矩阵是基础，但它只告诉你"是什么"。真正推动产品决策的是"so what"——即战略建议。比如"竞品 A 在某功能方面领先我们"是观察，"优先投入该功能以堵住竞品优势"才是可执行的战略建议。

### 学习要点

- ✅ 功能矩阵要诚实——全赢的矩阵没有可信度
- ✅ 最危险的竞品不一定是直接竞品，可能是用不同方式解决同样问题的
- ✅ 竞品分析的产出是"我们该怎么做"，不是"他们在做什么"

### ASCII 原理图

```
  竞品分类地图

  直接竞品（核心场景争夺）
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │  竞品 A  │ │  竞品 B  │ │  竞品 C  │
  │ 同一领域 │ │ 跨界进入 │ │ 价格竞争 │
  │ 区域优势 │ │ 技术优势 │ │ 下沉市场 │
  └──────────┘ └──────────┘ └──────────┘

        ┌─── 你的产品 ───────┐
        │  核心差异化优势    │
        │  定位 + 壁垒      │
        └─────────────────────┘

  相邻竞品                  替代方案
  (可能进入)                (完全不同)
  ┌──────────────┐          ┌──────────┐
  │ 大平台       │          │ 手工方式 │
  │ 延伸到你的   │          │ 传统工具 │
  │ 领域         │          │ Excel等  │
  └──────────────┘          └──────────┘
```

## 第4关：数据叙事

### 验证题

**问题**：同样的数据要给高层领导和产品团队看，最关键的区别是？

- A. 图表配色不同
- B. 数据精度不同
- C. 受众的决策场景不同，决定了叙事重点和数据密度（✅ 正确）
- D. 使用的图表类型不同

**解析**：数据叙事的第一步是"谁在看？要做什么决策？"高层领导关心宏观成果和战略方向（3-5 个核心数字就够），产品团队关心功能数据、漏斗和归因（需要详细下钻）。受众不同，数据密度、语言风格、叙事重点全部不同。

### 学习要点

- ✅ 数据叙事三段式：现状（10秒）→ 洞察（30秒）→ 行动（1分钟）
- ✅ 标题写洞察不写数据名（"XX指标领先"而非"各维度平均值"）
- ✅ Yellow 不是失败，是诚实的风险管理——比假 Green 强一百倍

### ASCII 原理图

```
  数据叙事三段式

  ┌──────────────────────────────┐
  │  现状 Situation              │
  │  ┌────┐ ┌────┐ ┌────┐ ┌────┐│
  │  │KPI1│ │KPI2│ │KPI3│ │KPI4││
  │  │🟢  │ │🟢  │ │🟡  │ │🟢  ││
  │  └────┘ └────┘ └────┘ └────┘│
  │  ← 10 秒看完                │
  ├──────────────────────────────┤
  │  洞察 Insight                │
  │  "XX 指标表现最突出           │
  │   但 YY 功能使用率在下降"     │
  │  ← 30 秒理解                │
  ├──────────────────────────────┤
  │  行动 Action                 │
  │  1. 推广成功经验到其他维度    │
  │  2. 调研使用率下降原因        │
  │  3. 下季度重点优化该功能      │
  │  ← 1 分钟知道该做什么        │
  └──────────────────────────────┘

  受众适配

  高层领导 ── 数据密度：低（3-5个数字）
              语言：结论先行，百分比和排名

  中层管理 ── 数据密度：中（对比表）
              语言：对比驱动，本部门 vs 整体

  产品团队 ── 数据密度：高（详细图表+下钻）
              语言：数据驱动，趋势和归因
```

## 第5关：数据价值看板

### 验证题

**问题**：设计 Dashboard 的第一步应该是？

- A. 选择合适的图表类型
- B. 收集所有可用的数据
- C. 明确受众要做什么决策（✅ 正确）
- D. 确定配色方案

**解析**：Dashboard 是决策工具，不是数据展示墙。第一步必须问"谁在看？要做什么决策？"不同受众需要的 KPI、图表、叙事全部不同。

### 学习要点

- ✅ Dashboard = 决策工具，不是数据展示墙
- ✅ 图表标题写洞察（"XX维度表现最佳"）而非数据名（"各维度平均值"）
- ✅ KPI 卡片三要素：数值 + 环比变化 + 状态颜色 = 10 秒理解

## 第6关：创建你的产品专属技能 — 毕业

### 验证题

**问题**：为什么实体消歧（Entity Disambiguation）是数据分析 Skill 的第一步？

- A. 因为它是技术规范要求的
- B. 因为同一个词在不同人嘴里含义不同，是数据查询出错的头号原因（✅ 正确）
- C. 因为消歧后查询速度更快
- D. 因为数据库设计需要实体定义

**解析**：当有人说"用户增长了10%"，他说的是哪类用户？当有人问"上个月的数据"，他指的是哪种数据口径？实体消歧不是技术细节，而是确保每个人查数据时问的是同一个问题。这比 SQL 语法错误更难发现，也更难修复。

### 学习要点

- ✅ 元技能 = 创造技能的技能，让团队成员无需重复学习
- ✅ 实体消歧是数据查询出错的头号原因
- ✅ 好的数据 Skill 让新人"读完就能正确查数据"
- ✅ Skill 需要持续迭代，不是一次性产出
