# 设计评审工具箱

为设计师提供 4 个核心评审工作流。基于 Anthropic 官方 Design 框架适配。

**与 /ui-ux-design 的关系：**
- `/ui-ux-design` = **生成**（配色/字体/风格 → 设计系统）
- `/design-review` = **评审**（设计稿 → 反馈/交付/无障碍检查）
- 两者互补，零重叠

## 启动流程

### 有参数时：自动识别意图

根据 `$ARGUMENTS` 中的关键词匹配功能：
- 含"评审""review""反馈""critique" → 设计评审
- 含"交付""handoff""标注""开发" → 开发交付文档
- 含"无障碍""accessible""WCAG""a11y" → 无障碍审查
- 含"文案""copy""提示""空状态""错误" → UX 文案
- 无法识别 → 弹出选择菜单

### 无参数时：弹出选择菜单

使用 AskUserQuestion 提供 4 个选项：

| 选项 | 说明 |
|------|------|
| 设计评审 | 从 5 个维度评审设计稿，输出结构化反馈 |
| 开发交付文档 | 生成开发所需的标注、状态、边界情况文档 |
| 无障碍审查 | 按 WCAG 2.1 AA 标准检查无障碍合规 |
| UX 文案 | 撰写界面文案（错误提示/空状态/引导/按钮） |

## 功能 1：设计评审

**读取** `references/critique.md` 获取完整框架。

**收集设计信息：**
1. 请描述你的设计（文字描述、截图路径、或口述方案都可以）
2. 这是什么阶段的设计？（选择：概念草图 / 中保真线框 / 高保真视觉稿 / 已上线页面）
3. 有特别想关注的方面吗？（选择：整体评审 / 可用性 / 视觉层级 / 一致性 / 无障碍）

**输出结构：**

```
## 设计评审：[设计名称]

### 整体印象
[2 秒内的第一反应：视觉焦点在哪？情绪感受？目的是否清晰？]

### 可用性
| 发现 | 严重度 | 建议 |
|------|--------|------|
| ... | 🔴/🟡/🟢 | ... |

### 视觉层级
[阅读顺序分析、重点元素是否突出、留白使用]

### 一致性
| 元素 | 问题 | 建议 |
|------|------|------|
| ... | ... | ... |

### 无障碍
[对比度、触控区域、可读性速查]

### 做得好的地方
[至少 2-3 个正面反馈]

### Top 3 优先改进
1. ...
2. ...
3. ...
```

**输出后追问：** "需要我针对某个问题深入分析吗？或者生成这个设计的开发交付文档？"

## 功能 2：开发交付文档

**读取** `references/handoff.md` 获取完整框架。

**收集信息：**
1. 请描述要交付的设计（页面/组件/功能）
2. 目标技术栈？（选择：Web(React/Vue) / iOS / Android / 跨平台 / 不确定）
3. 需要覆盖的内容？（多选：布局标注 / 设计 Token / 组件状态 / 响应式 / 动效 / 全部）

**输出结构：**

```
## 交付文档：[功能/页面名称]

### 概述
[功能说明 + 设计意图]

### 布局
[网格、间距、断点]

### 设计 Token
| Token | 值 | 用途 |
|-------|---|------|

### 组件清单
| 组件 | 变体 | 属性 | 备注 |
|------|------|------|------|

### 状态与交互
| 元素 | 状态 | 行为 |
|------|------|------|

### 响应式行为
| 断点 | 变化 |
|------|------|

### 边界情况
[空状态、长文本、加载中、错误、极端数据]

### 动效
| 元素 | 触发 | 动画 | 时长 | 缓动 |
|------|------|------|------|------|

### 无障碍备注
[焦点顺序、ARIA、键盘操作、屏幕阅读器]
```

## 功能 3：无障碍审查

**读取** `references/accessibility.md` 获取完整框架。

**收集信息：**
1. 请描述要审查的设计/页面
2. 审查标准？（选择：WCAG 2.1 AA（推荐） / WCAG 2.1 AAA / 快速检查）
3. 重点关注？（选择：全面审查 / 视觉对比度 / 键盘导航 / 屏幕阅读器 / 触控适配）

**输出结构：**

```
## 无障碍审查：[页面名称]
标准：WCAG 2.1 AA | 日期：[今天]

### 摘要
| 严重度 | 数量 |
|--------|------|
| 🔴 严重 | X |
| 🟡 中等 | X |
| 🟢 建议 | X |

### 可感知（Perceivable）
| 标准 | 发现 | 严重度 | 建议 |
|------|------|--------|------|

### 可操作（Operable）
| 标准 | 发现 | 严重度 | 建议 |

### 可理解（Understandable）
| 标准 | 发现 | 严重度 | 建议 |

### 健壮性（Robust）
| 标准 | 发现 | 严重度 | 建议 |

### 颜色对比度检查
| 元素 | 前景色 | 背景色 | 对比度 | 达标？ |

### 优先修复清单
1. ...
2. ...
3. ...
```

## 功能 4：UX 文案

**读取** `references/ux-copy.md` 获取完整框架。

**收集信息：**
1. 需要什么类型的文案？（选择：按钮/CTA / 错误提示 / 空状态 / 确认对话框 / 引导文案 / 工具提示）
2. 请描述使用场景和上下文
3. 品牌语气？（选择：专业正式 / 友好亲切 / 轻松活泼 / 简洁克制）

**输出结构：**

```
## UX 文案：[场景]

### 推荐文案
[主要推荐的文案方案]

### 备选方案
| 方案 | 文案 | 语气 | 适合场景 |
|------|------|------|---------|
| A | ... | 正式 | B 端产品 |
| B | ... | 亲切 | C 端产品 |
| C | ... | 简洁 | 移动端 |

### 设计理由
[为什么推荐这个方案]

### 本地化提示
[翻译/多语言需要注意的点]
```

## 通用规则

- 所有输出使用中文
- 反馈要具体——"这里不太好"不是反馈，"标题和正文对比度不足(2.8:1)，建议提高到 4.5:1 以上"才是
- 先说做得好的，再说需要改进的
- 严重度标记要客观：🔴 影响功能使用 / 🟡 影响体验 / 🟢 优化建议
- 评审匹配设计阶段——不要在草图阶段挑像素级问题
- WCAG 标准号保留英文编号（如 1.4.3），说明用中文
